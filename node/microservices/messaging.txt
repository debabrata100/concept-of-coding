# Messaging in Microservices
## What is Messaging?
In microservices architecture, services need to communicate with each other. Messaging is an asynchronous communication mechanism where services send messages to each other using a message broker or queue.

Instead of direct (synchronous) API calls (like REST), a service can send a message to a queue or topic, and the other service can consume it later. This makes the system more resilient, loosely coupled, and scalable.

Benefits of Messaging
✅ Decoupling: Services don’t need to know each other's location or status.

🔁 Asynchronous processing: Services can process messages independently.

💥 Resilience: If a consumer service is down, messages can be held until it's back.

📈 Scalability: Multiple consumers can scale horizontally to process messages.

Common Messaging Patterns
# Point-to-Point (Queue):
One sender, one receiver.
Used for task queues.

# Publish-Subscribe (Topic):
One sender, multiple receivers.
Used for events/notifications.

# Event-Driven Architecture (EDA):
Services emit events when state changes.
Other services subscribe to those events and react.
