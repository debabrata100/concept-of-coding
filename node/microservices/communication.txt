# Two Main Categories of Communication
* Synchronous Communication
HTTP/REST
gRPC
GraphQL
Thrift
WebSockets (for streaming)

* Asynchronous Communication
Message Brokers: RabbitMQ, Kafka, NATS, Amazon SQS/SNS, Google Pub/Sub
Event Streams: Kafka, Pulsar
Serverless Triggers: AWS Lambda, EventBridge
* Other Communication Methods
- Service Mesh (e.g., Istio, Linkerd)
Abstracts service-to-service communication
Handles retries, timeouts, TLS, load balancing
Use Case: Large-scale microservices where you need advanced routing and security.
- Webhooks
One service calls another when an event occurs (push instead of poll)
Use Case: Payment gateways, integrations

* When to Use What?

Scenario	                                Best Choice
Request-response, real-time	                REST or gRPC
Background jobs, retries	                RabbitMQ, SQS
Multiple consumers need same message	    Kafka, Pub/Sub
Event-based system	                      Kafka, EventBridge
Large data streams	                      Kafka, Pulsar
Aggregating APIs	                            GraphQL
Complex service communication	                Service Mesh (Istio)

A. HTTP/REST APIs
Most common
Easy to understand and debug
Text-based (JSON/XML)
Use Case: CRUD operations, synchronous workflows.
GET /users/123
✅ Pros:
Simple
Human-readable
Tools like Postman, Swagger, etc.
❌ Cons:
Tight coupling
Failures propagate (if service is down, request fails)
Slower under load
B. gRPC (Google Remote Procedure Call)
Uses HTTP/2
Binary data via Protocol Buffers (protobuf)
Faster and smaller payloads
Use Case: High-performance internal service communication.
✅ Pros:
Lightweight, fast
Code generation for multiple languages
Streaming supported
❌ Cons:
Steeper learning curve
Harder to debug than REST
Not ideal for external APIs
C. GraphQL
Allows clients to ask for exactly the data they need.
Reduces over-fetching/under-fetching.
Use Case: Aggregating data from multiple microservices in a single query.
✅ Pros:
Efficient, flexible
Good for frontends
❌ Cons:
Adds complexity
Not great for write-heavy systems

A. Message Queues (Point-to-Point)
One producer → one consumer
Tasks or job queues
Use Case: Background processing, retry logic, task queues.
✅ Pros:
Loose coupling
Retry on failure
Throttle message load
❌ Cons:
Messages can pile up
Order may not be guaranteed
B. Publish-Subscribe
One producer → multiple subscribers
Messages broadcast to all
Use Case: Notifications, user activity logs, cache invalidation.
✅ Pros:
Decoupled services
Scalability
❌ Cons:
Harder to guarantee delivery to all consumers

C. Event-Driven Architecture
Services emit events when things happen.
Other services subscribe and react.
Use Case: Order management, workflows, microservices choreography.
✅ Pros:
Reactive and decoupled
Scalable
Easier to evolve independently
❌ Cons:
Hard to trace and debug
Requires strong observability

D. Streaming
Continuous flow of data (Kafka, Pulsar)
Consumers process data in real-time or near real-time
Use Case: Analytics, monitoring, financial data processing.
✅ Pros:
Real-time insights
Scalable consumers
❌ Cons:
Complex architecture
Requires careful management

# Interview Tip:
When asked, explain not just the tool, but why you would pick it. Link your choice to 
scalability, latency, resilience, and team experience.

MOCK INTERVIEW QUESTIONS (with expected answers)
🔹 Q1: What are the different ways microservices can communicate?
A:
Microservices can communicate in synchronous and asynchronous ways:
Synchronous:
REST APIs (HTTP)
gRPC
GraphQL
Asynchronous:
Message queues (e.g., RabbitMQ, SQS)
Pub/Sub (Kafka, SNS, Google Pub/Sub)
Event-driven architecture (Kafka, EventBridge)
Streaming (Kafka, Pulsar)
🔹 Q2: When would you use asynchronous communication over synchronous?
A: Use asynchronous communication when:
Services don't need an immediate response.
You want loose coupling between services.
You’re dealing with background processing or event-driven workflows.
You need resilience, e.g., retrying or buffering failures.
Example: A payment service emits a PaymentCompleted event, and the invoice service picks it up to generate an invoice — no need to block the payment flow.

🔹 Q3: What are the trade-offs of using gRPC over REST?
A:
Aspect	    gRPC	                    REST
Speed	    Faster (binary)	            Slower (text-based)
Schema	    Strongly typed (Protobuf)	Weakly typed (JSON)
Tooling	    Auto-code generation	    Easy debugging
Use case	Internal, high-performance	Public APIs, interoperability

I would choose gRPC for internal microservice-to-microservice calls and REST for external APIs.

🔹 Q4: How would you design an order management system using event-driven architecture?
🧱 Components:
Order Service: Places an order and emits OrderCreated.
Inventory Service: Listens to OrderCreated, reserves stock.
Notification Service: Sends confirmation.
Billing Service: Charges the customer.

📡 Communication:
Use Kafka or EventBridge to emit and consume events like:
OrderCreated
InventoryReserved
PaymentProcessed
📊 Benefits:
Loose coupling
Scalability
Each service can evolve independently
