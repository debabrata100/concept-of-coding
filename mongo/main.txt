# Relationships in MongoDB:
here are two main strategies to model relationships in MongoDB:

1. References (Normalization)
This is what we covered earlier using ObjectId + populate().
2. Embedding (Denormalization)
Instead of referencing, you embed related data inside the document.

Feature	            References	                Embedding
Structure	        Separate collections	    Nested inside parent document
Speed	            Slower read (needs join)	Faster read (one query)
Flexibility	        Better if many-to-many	    Best for one-to-few
Update overhead	    Easier updates	            Update whole document

# Example: Embed Posts inside User
* User Schema with Embedded Posts
const mongoose = require('mongoose');
const { Schema, model } = mongoose;

const postSchema = new Schema({
  title: String,
  content: String,
  createdAt: { type: Date, default: Date.now }
});

const userSchema = new Schema({
  name: String,
  email: String,
  posts: [postSchema]  // 👈 Embedded subdocuments
});

const User = model('User', userSchema);
# Create User with Posts
await User.create({
    name: "Alice",
    email: "alice@example.com",
    posts: [
      { title: "First Post", content: "Hello!" },
      { title: "Second Post", content: "How are you?" }
    ]
  });

# Fetch User with Posts
const user = await User.findOne({ email: "alice@example.com" });
console.log(user.posts); // Already included, no populate needed

# When to Use Embedding?
✅ Use embedding when:

Related data is closely tied to the parent.

You don’t need to query posts independently.

There’s a limited number of subdocuments per parent (MongoDB document size limit is 16MB).

❌ Avoid embedding if:

Subdocuments grow unbounded (e.g., thousands of comments/posts).

You need to frequently update subdocuments independently.

# Hybrid Model (Bonus Tip)
you can also use a hybrid approach:
Store essential data of posts inside user (title, id).
Keep full post data in a separate posts collection.
userSchema = new Schema({
    name: String,
    posts: [
      {
        _id: Schema.Types.ObjectId,
        title: String
      }
    ]
  });
  Approach	        Pros	                                        Use When...
  ObjectId refs	    Normalized, flexible, good for many-to-many	    Posts are a separate entity
  Embedded docs	    Faster reads, one query	                        Posts are small + tightly coupled
  Hybrid	        Balance of both	                                Need partial data fast, full data sometimes
