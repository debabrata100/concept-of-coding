When should you <link rel=”preload”> vs <link rel=”prefetch”>?
Tip: Preload resources you have high-confidence will be used in the current page. Prefetch resources likely to be used for future navigations across multiple navigation boundaries.

Preload is an early fetch instruction to the browser to request a resource needed for a page (key scripts, Web Fonts, hero images).

Prefetch serves a slightly different use case — a future navigation by the user (e.g between views or pages) where fetched resources and requests need to persist across navigations. If Page A initiates a prefetch request for critical resources needed for Page B, the critical resource and navigation requests can be completed in parallel. If we used preload for this use case, it would be immediately cancelled on Page A’s unload.

Between preload and prefetch, we get solutions for loading critical resources for the current navigation _or_ a future navigation.

What is the caching behavior for <link rel=”preload”> and <link rel=”prefetch”>?
Chrome has four caches: the HTTP cache, memory cache, Service Worker cache & Push cache. Both preload and prefetched resources are stored in the HTTP cache.

When a resource is preloaded or prefetched it travels up from the net stack through to the HTTP cache and into the renderer’s memory cache. If the resource can be cached (e.g there’s a valid cache-control with valid max-age), it is stored in the HTTP cache and is available for current and future sessions. If the resource is not cacheable, it does not get stored in the HTTP cache. Instead, it goes up to the memory cache and stays there until it gets used.

HTTP Cache:

The HTTP cache refers to the caching mechanism implemented in web browsers and intermediary proxy servers.
When a browser requests a resource (like an HTML page, CSS file, or image) from a server, it can cache that resource locally based on caching directives sent by the server in the HTTP response headers.
These directives include Cache-Control, Expires, ETag, and Last-Modified.
The cache allows the browser to store a copy of the resource locally, which can be reused for subsequent requests, reducing the need to fetch the resource from the server again.
HTTP caching can significantly improve website performance by reducing latency and bandwidth usage.
Memory Cache:

Memory caching refers to storing frequently accessed data in the computer's main memory (RAM) for faster access.
In web development, memory caching can be utilized by web servers, application servers, or programming languages to store frequently accessed data structures or objects in memory.
Memory caching is much faster than fetching data from disk storage because RAM access times are significantly shorter.
Popular memory caching systems include Memcached and Redis, which are used to cache frequently accessed data in distributed systems and web applications.
Service Worker Cache:

Service workers are JavaScript files that run in the background of web browsers, separate from the main browser thread.
Service workers have the ability to intercept and handle network requests made by web pages.
One of the features of service workers is the ability to cache web resources locally, independent of the browser's HTTP cache.
Service worker caching allows developers to implement advanced caching strategies, such as offline support for web applications (Progressive Web Apps) and custom caching logic.
Service worker caches are typically used to store assets like HTML, CSS, JavaScript, and images needed for offline functionality or to improve performance by serving cached resources directly from the service worker.
Server Push Cache:

Server push caching refers to caching mechanisms employed by servers to push resources to clients (typically web browsers) before they are requested.
This technique is often used in conjunction with HTTP/2, which supports server push.
With server push caching, the server can anticipate which resources the client will need based on the initial request and proactively push those resources to the client's cache.
By pushing resources to the client in advance, server push caching can reduce latency and improve page load times, especially for resources like CSS, JavaScript, and images.
Server push caching requires support from both the server and the client (browser) and can be used to complement other caching mechanisms like HTTP caching and service worker caching.
